\section{Benchmark des technologies}
\label{sec:benchmark}
%% ============================================================

Ce paragraphe présente les comparaisons sur lesquelles reposent les choix technologiques du framework.

\subsection{Outil ETL}

Le choix de l'outil ETL est engagé car il va conditionner la façon dont les pipelines d'import sont réalisés, maintenus et générés automatiquement par IA. Les processus ETL sont un des domaines majeurs de ce type d'architectures d'intégration de données~\cite{vassiliadis2009}. Trois solutions ont ainsi été testées :

\begin{itemize}
    \item \textbf{Apache Hop~:} plateforme ETL open source (licence Apache 2.0), fork modernisé de Kettle fondé en 2020 par les auteurs originaux. Les pipelines sont décrits dans des fichiers XML (\texttt{.hpl}), exécutables en ligne de commande via \texttt{hop-run.sh}.
    \item \textbf{Talend Open Studio~:} outil ETL sous un modèle freemium. Son interface graphique génère du code Java compilé (.jar). Le versionnement est complexe puisque chaque modification génère l'ensemble du code source.
    \item \textbf{Apache NiFi~:} plateforme de data flow temps réel, conçue pour le routage et la distribution de données en streaming (Kafka, IoT, API). Fonctionne comme un serveur web permanent avec une interface de configuration.
\end{itemize}

Les critères de comparaison retenus sont les suivants :

\begin{itemize}
    \item \textbf{Format des pipelines~:} format dans lequel les pipelines sont stockés sur disque. Un format texte structuré (XML) sera critère essentiel pour notre projet car il permet la génération automatique par un LLM et le versionnement ligne à ligne dans Git.
    \item \textbf{Généreable par LLM~:} capacité d'un modèle de langage à produire un pipeline valide à partir d'instructions. Cela nécessite qu'il s'agisse d'un format texte, lisible, avec une structure prévisible.
    \item \textbf{Pattern upsert natif~:} un composant built-in permettant de vérifier avant appel entre INSERT/UPDATE la préexistence d'un enregistrement, sans nécessiter du scripting custom.
    \item \textbf{Exécution CLI sans serveur~:} une exécution d'un pipeline par une unique commande terminale, à condition qu'un serveur ne soit pas requis pour fonctionner sans interruption.
\end{itemize}

\textbf{Choix arbitré~:} Apache Hop. Le critère déterminant est bien la \textbf{générabilité par LLM}~: les pipelines étant des fichiers XML dont la structure est prévisible, un agent IA peut les lire comme contexte et les générer. Aucune autre solution n'en permet autant~: Talend produit du bytecode Java non exploitable, tandis que NiFi met sa configuration dans un stockage interne non orienté vers la production programmatisée. De surcroît, Hop offre nativement le pattern upsert (composant Insert/Update) et l'exécution CLI, deux prérequis fondamentaux à notre architecture.

\subsection{Agent IA de génération de code}

Le framework du projet sera un agent IA, capable de générer du code structuré (notamment XML Apache Hop, JSON) à partir d'instructions écrites en langage naturel par l'utilisateur. Les grands modèles de langage ont montré leur capacité, désormais avérée, à générer du code structurel en se basant sur des exemples~\cite{brown2020} et donc à créer un agent d'assistance au développement~\cite{chen2023}. On peut notamment citer :

\begin{itemize}
    \item \textbf{Claude Code (Anthropic)~:} agent IA de génération de code développé par Anthropic, basé sur le modèle Claude~\cite{anthropic_claude}.
    \item \textbf{Codex CLI (OpenAI)~:} agent open source (MIT), tourne localement en CLI en étant basé sur GPT-5.5~\cite{openai_codex}.
    \item \textbf{Gemini CLI (Google)~:} agent open source (Apache 2.0), basé sur Gemini 3.1 Pro~\cite{google_gemini_cli}.
\end{itemize}

\textbf{Choix retenu~:} Claude Code. Entre les trois agents comparables en performance sur les benchmarks, le choix s'est porté sur une dimension pragmatique~: l'entreprise avait déjà souscrit un abonnement en cours auprès d'Anthropic, et il y avait au démarrage du projet une certaine familiarité avec l'écosystème Claude.

\textbf{Backend (FastAPI/Python)~:} Python a été choisi car le backend doit surtout orchestrer des appels CLI (lancer Claude via \texttt{subprocess}, spawner des PTY, exécuter \texttt{hop-run.sh}). Python offre une gestion native des processus système et un prototypage rapide, adapté à un projet de recherche~\cite{fastapi_doc}.

\textbf{Frontend (React + xterm.js)~:} le choix de React~\cite{react_doc} repose avant tout sur sa communauté très active, qui a déjà développé un grand nombre de composants prêts à l'emploi. L'interface de l'application n'a pas vocation à être technique~: elle doit rendre le flux d'import accessible à un utilisateur non développeur, et l'écosystème React facilite cela bien plus qu'un framework comme Angular, qui impose une courbe d'apprentissage plus lourde. En ce qui concerne la bibliothèque xterm.js~\cite{xtermjs} utilisée pour l'émulation terminal dans le navigateur, via communication WebSocket~\cite{websocket_rfc}, elle est indépendante du framework.

\textbf{PostgreSQL~:} imposé par Axelor Open Suite, seule base supportée par l'ERP : pas de choix à faire ici.

\section{Architecture globale de la solution}
\label{sec:archi-globale}
%% ============================================================

Le cadre suit un parcours linéaire allant du fichier source formaté CSV à l'insertion en base de données~:

\begin{center}
\textbf{CSV} $\longrightarrow$ \textbf{Agent IA} (analyse + manifeste) $\longrightarrow$ \textbf{Agent IA} (génération de pipeline) $\longrightarrow$ \textbf{Apache Hop} (exécution) $\longrightarrow$ \textbf{PostgreSQL}
\end{center}

Le processus est divisé en trois étapes~:

\begin{enumerate}
    \item \textbf{Analyse et mapping~:} le consultant uploade un fichier CSV dans l'application Mapper. L'agent IA analyse la structure, identifie les colonnes et assure les dépendances FK en interrogeant la base cible. Enfin, il produit le \textbf{manifeste JSON} du mapping complet.
    \item \textbf{Génération des pipelines~:} à partir du manifeste, l'agent IA génère les fichiers XML Apache Hop (\texttt{.hpl}) à partir de pipelines de référence comme modèles.
    \item \textbf{Exécution~:} les pipelines générés sont exécutés par le moteur Apache Hop via \texttt{hop-run.sh}, qui lit le fichier CSV, applique les transformations, et insère les données dans PostgreSQL.
\end{enumerate}

\subsection{Le manifeste JSON en tant qu'artefact intermédiaire}

Le manifeste JSON est un fichier pérenne sur disque entre l'étape d'analyse et l'étape de génération. Il formalise~: la table cible, le mapping des colonnes, les stratégies de résolution FK, l'ordre des entités à importer et les métadonnées de transformation. Cette pérennisation permet de~:
\begin{itemize}
    \item Valider le mapping avec le consultant avant génération des pipelines ;
    \item Versionner le résultat de l'analyse avec Git ;
    \item Régénérer les pipelines sans relancer l'analyse si seul le format de sortie change.
\end{itemize}

\subsection{Composants du système}

\begin{itemize}
    \item \textbf{Application Axelor Import Studio~:} interface web (React + FastAPI) offrant terminal de conversation et actions post-mapping (génération, exécution, consultation des résultats).
    \item \textbf{Compétences en IA~:} prompts structurés guidant l'agent à travers l'analyse du CSV et la création des pipelines.
    \item \textbf{Manifeste JSON~:} fichier décrivant le mapping global d'une entité.
    \item \textbf{Pipelines Apache Hop~:} fichiers XML (\texttt{.hpl}) mettant en œuvre les patterns d'import (upsert, junction, post-SQL).
    \item \textbf{Workflows Apache Hop~:} fichiers XML (\texttt{.hwf}) orchestrant le lancement séquentiel des pipelines.
\end{itemize}

\section{Sécurité et conformité RGPD}
\label{sec:securite}
%% ============================================================

\subsection{Mesures de sécurité existantes}

\begin{itemize}
    \item \textbf{Accès direct PostgreSQL~:} les identifiants de la base cible sont renseignés dans l'interface au démarrage de la session et stockés côté serveur backend. Aucune sollicitation API vers l'application Axelor n'est réalisée.
    \item \textbf{Sessions temporaires~:} chaque session est identifiable par un UUID aléatoire. L'outil ne possède pas de système d'authentification utilisateur, car il est supposé être utilisé en interne, dans un environnement contrôlé.
    \item \textbf{Authentification Claude~:} l'accès à Claude se fait via le CLI (\texttt{claude login}) avec le compte utilisateur Anthropic. Aucune clé API n'est gérée ni stockée par l'application.
\end{itemize}

\subsection{Conformité RGPD et données personnelles}

L'agent IA (Claude) exploite les fichiers CSV fournis par l'utilisateur dans le cadre de son interaction. Si ces fichiers contiennent des données personnelles (noms, adresses, emails de clients réels), celles-ci sont adressées au modèle hébergé par Anthropic.

\textbf{Limite aujourd'hui~:} l'outil ne permet pas d'anonymiser automatiquement les données avant leur envoi au modèle. Il appartient donc à l'utilisateur de~:
\begin{itemize}
    \item Traiter avec des fichiers d'entrée anonymisés ou fictifs dans le cadre de la phase de mapping ;
    \item Travailler sur une base test (et non productive) ;
    \item Ne pas exposer de données personnelles réelles à l'agent IA.
\end{itemize}

Cette politique est conforme à l'usage qui est fait en environnement de développement, pour lequel les données test sont généralement synthétiques, alors que l'anonymisation automatique des données en amont devrait constituer un axe de progrès à l'avenir.
