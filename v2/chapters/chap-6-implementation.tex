Les dossiers principaux sont~:
\begin{itemize}
    \item \texttt{datasets/} — fichiers CSV source organisés par entité ;

\subsubsection{Rapports d'erreurs}

Chaque pipeline génère automatiquement un rapport d'erreurs au format CSV qui se trouve dans le dossier \texttt{reports/}. Quand un enregistrement ne peut pas être importé (FK non résolue, impossibilité de retrouver import\_id, valeur inappropriée), il est routé vers le fichier erreurs correspondant et non pas tranquillement ignoré.

Dans le rapport figure~: l'identifiant de l'enregistrement source, les champs pertinents (nom, code), et un message qualifiant la cause de l'échec. L'information permet au consultant de corriger la donnée source, et de relancer l'import — l'idempotence garantit que seuls les enregistrements corrigés pourront être réinsérés.

\subsubsection{Skill analyze-and-map-entity}

Ce skill, composé de 476 lignes et six fichiers de référence, permet à l'agent de traiter un fichier CSV et de générer un manifeste au format JSON. Les étapes sont les suivantes~:

\paragraph{Étape 0 : Connexion à la base de données} L'agent commence par vérifier la connexion à PostgreSQL avec les identifiants fournis. Si la connexion est réussie, toutes les vérifications des clés étrangères suivantes seront effectuées en interrogeant la base de données réelle. Dans le cas contraire, l'agent se met en mode dégradé et s'appuie uniquement sur les réponses fournies par l'utilisateur.

\paragraph{Étape 1 : Lecture et classification des colonnes} L'agent lit le fichier CSV (en prenant en compte les en-têtes et les vingt premières lignes) tout en interrogeant simultanément le schéma de la table cible via \texttt{information\_schema} pour obtenir les colonnes, les types et les contraintes des clés étrangères. Il catégorise ensuite chacune des colonnes en croisant trois signaux~:
\begin{itemize}
    \item \textbf{Notation pointée} (confiance forte)~: \texttt{productFamily.importId} renvoie directement à la table cible et donc à la stratégie de lookup ;
    \item \textbf{Pattern du nom} (confiance modérée)~: un suffixe \texttt{Code}, \texttt{Id} ou \texttt{Name} indique une FK ;
    \item \textbf{Échantillonnage des valeurs} (vérification)~: faible cardinalité, valeurs courtes et homogènes indiquent une FK.
\end{itemize}
Pour les cas ambigus, l'agent pose une question à l'utilisateur en lui proposant des options.

\paragraph{Étape 2 : Vérification en base} Pour chaque FK identifiée, l'agent exécute une requête SQL pour vérifier que les valeurs du CSV existent dans la table référencée. Les FK sont regroupées par table de référence (ex~: \texttt{saleCurrency} et \texttt{purchaseCurrency} → une seule requête sur \texttt{base\_currency}). Trois résultats sont possibles~: tout trouvé (preloaded), partiellement trouvé, rien trouvé.

\paragraph{Étape 3 : Résolution interactive} L'agent questionne l'utilisateur sur la stratégie à adopter pour chaque FK non résolue~:
\begin{itemize}
    \item \textbf{Fichier séparé}~: l'utilisateur fournit un CSV spécifique (l'agent applique récursivement le même processus pour ce fichier) ;
    \item \textbf{Embedded (create automatically)}~: un pré-pipeline sera produit pour insérer les valeurs manquantes avant l'import principal ;
    \item \textbf{Skip}~: le champ FK restera NULL.
\end{itemize}

\paragraph{Étape 4 : Confirmation} L'agent propose un tableau récapitulatif de toutes les résolutions FK et demande confirmation avant la phase d'écriture.

\paragraph{Étape 5 : Écriture du manifeste} L'agent produit le fichier \texttt{manifest.json} et l'écrit sur disque au choix de l'utilisateur.

Le manifeste est le fichier intermédiaire qui formalise le résultat de l'analyse, il est donc nécessaire qu'il contienne toutes les informations nécessaires pour permettre au skill de génération de produire les pipelines sans ambiguïté. Chaque section d'entité dans le manifeste précise~:
\begin{itemize}
    \item la table cible et le fichier source ;
    \item le mapping colonne par colonne (source CSV → colonne DB) ;
    \item les résolutions FK avec la stratégie de lookup et le comportement en cas d'absence ;
    \item les champs calculés, tables secondaires, constantes d'insertion ;
    \item l'ordre d'import permettant de respecter les dépendances entre entités.
\end{itemize}

C'est ce fichier que consomme le skill \textit{generate-pipeline} pour produire le XML Apache Hop. Il est structuré de la façon suivante~:

\begin{verbatim}
{
  "primary_entity": "product",
  "import_order": ["product_family", "product_category", "product"],
  "pre_pipelines": [...],
  "entities": {
    "product": {
      "target_table": "base_product",
      "pipeline_name": "csv_to_db_products",
      "write_mode": "upsert",
      "source": { "file": "datasets/product/products.csv", "delimiter": ";" },
      "column_mappings": [
        { "source": "name", "target": "name", "type": "string" },
        { "source": "salePrice", "target": "sale_price", "type": "decimal" }
      ],
      "fk_resolutions": [
        {
          "source_column": "productFamily.importId",
          "target_fk_field": "product_family_id",
          "db_column": "product_family",
          "ref_table": "base_product_family",
          "strategies": ["by_import_id"],
          "fk_source": "separate_entity",
          "on_miss": "set_null"
        }
      ],
      "computed_fields": [...],
      "secondary_tables": [...],
      "change_detection_fields": ["name", "code"],
      "insert_constants": { "is_model": false }
    }
  }
}
\end{verbatim}

Chaque champ guide directement la génération du pipeline~:
\begin{itemize}
    \item \texttt{write\_mode} détermine le pattern (upsert avec UPDATE, ou insert-only sans) ;
    \item \texttt{column\_mappings} génère le bloc \texttt{Select Values} (renommage CSV → DB) et le \texttt{Table Output} ;
    \item \texttt{fk\_resolutions} génère un \texttt{DB Lookup} par FK, avec la table, la clé et le comportement en cas d'absence ;
    \item \texttt{change\_detection\_fields} génère le filtre ``has changes'' (comparaison avant/après pour l'UPDATE) ;
    \item \texttt{import\_order} détermine l'enchaînement dans le workflow.
\end{itemize}

Le manifeste supporte également des patterns avancés~:

\paragraph{Pattern embedded (création automatique de références manquantes)}
Lorsqu'une colonne CSV contient directement le nom d'une référence (ex~: \texttt{unit = "Jour ouvré"}) au lieu d'un ID, le manifeste utilise \texttt{fk\_source: "embedded"} avec un bloc \texttt{pre\_pipeline\_config}~:

\begin{verbatim}
{
  "source_column": "unit",
  "ref_table": "base_unit",
  "lookup_field": "name",
  "fk_source": "embedded",
  "pre_pipeline_config": {
    "name": "create_missing_units",
    "subdir": "product/units",
    "sequence": "base_unit_seq"
  }
}
\end{verbatim}

Cela génère un pré-pipeline qui extrait les valeurs uniques du CSV, vérifie leur existence en base, et insère les manquantes avant l'import principal.

\paragraph{Pattern parse\_config (adresses embarquées)}
Pour les entités dont une colonne contient une adresse en texte libre (``70 RUE PASTORELLI | 06000 NICE | FRANCE''), le manifeste inclut un bloc \texttt{parse\_config} qui décrit le parsing~:

\begin{verbatim}
"parse_config": {
  "type": "regex_split",
  "source_field": "address",
  "pattern": "^([^|]+)\\|\\s*(\\d+)\\s+([^|]+)\\|\\s*(.+)$",
  "groups": [
    { "name": "addr_street" },
    { "name": "addr_zip" },
    { "name": "addr_city" },
    { "name": "addr_country" }
  ],
  "pre_dedup": { "sort_field": "address", "unique_field": "address" }
}
\end{verbatim}

Cela génère un pipeline insert-only avec~: tri + déduplication, RegexEval pour splitter l'adresse, puis insertion.

\paragraph{Pattern junction\_pipelines (tables M2M)}
Les relations Many-to-Many sont décrites au niveau racine du manifeste dans \texttt{junction\_pipelines[]}~:

\begin{verbatim}
"junction_pipelines": [{
  "name": "link_partner_address",
  "source": { "file": "datasets/partners/partnerAddresses.csv" },
  "side_a": { "csv_column": "partner_importId", "ref_table": "base_partner" },
  "side_b": { "csv_column": "address_importId", "ref_table": "base_address" },
  "junction_table": "base_partner_address",
  "sequence": "base_partner_address_seq"
}]
\end{verbatim}

Cela génère un pipeline qui résout les deux côtés par lookup, vérifie l'absence de doublon (idempotence), puis insère dans la table de jonction.

\subsubsection{Skill generate-pipeline}

Ce skill (1891 lignes + 3 fichiers de référence) permet de convertir un manifeste JSON en fichiers XML à l'aide d'Apache Hop. Il se déroule en cinq étapes~:

\begin{enumerate}
    \item \textbf{Étape 0 : Chemins} l'utilisateur fournit le chemin du manifeste et le répertoire où les fichiers générés seront enregistrés.
    \item \textbf{Étape 1 : Lecture du manifeste} cette étape extrait les informations d'importation, les sections d'entité (table cible, colonnes, clés étrangères, champs calculés), ainsi que les pipelines de jonction et post-traitement.
    \item \textbf{Étape 2 : Lecture des références} les pipelines de référence existants sont analysés pour reproduire parfaitement la structure XML. Trois fichiers de référence fournissent les blocs XML pour chaque type de transformation, les conventions essentielles (ex~: la valeur du transform \texttt{Constant} doit être placée dans \texttt{<nullif>} et non dans \texttt{<value>}), ainsi que l'architecture standard des chaînes de transformations.
    \item \textbf{Étape 3 : Génération} pour chaque entité dans l'ordre d'importation, un fichier \texttt{.hpl} est créé en suivant le modèle approprié~:
    \begin{itemize}
        \item \textbf{Upsert~:} CSV → Select Values → Filter → FK Lookups → Self-lookup → Filter is new → INSERT/UPDATE ;
        \item \textbf{Junction~:} CSV → Lookup côté A → Lookup côté B → Vérification d'idempotence → INSERT jonction ;
        \item \textbf{Post-SQL~:} RowGenerator → ExecSQL (requête SQL directe).
    \end{itemize}
    \item \textbf{Étape 5 : Vérification} chaque fichier est vérifié pour garantir qu'il s'agit d'un XML valide, et le nombre total de fichiers générés est rapporté.
\end{enumerate}

\subsubsection{Skill generate-workflow}

Ce skill (289 lignes + 1 fichier de référence) produit le workflow orchestrateur (\texttt{.hwf}) à partir du manifeste. Il enchaîne les actions dans cet ordre~:

\begin{enumerate}
    \item \textbf{Preflight checks~:} vérification de l'existence du fichier CSV source et test de la connexion BD. En cas d'échec → Abort.
    \item \textbf{Pré-pipelines~:} exécute les pipelines de création de références manquantes (ex~: \texttt{create\_missing\_units}) identifiés par les FK avec \texttt{fk\_source: "embedded"}.
    \item \textbf{Import order~:} exécute séquentiellement chaque pipeline d'entité dans l'ordre topologique. Échec → Abort.
    \item \textbf{Post-pipelines~:} exécute les pipelines de calcul (totaux, taxes, champs dérivés).
    \item \textbf{Junction pipelines~:} exécute les pipelines de liaison M2M.
    \item \textbf{Success~:} action finale si tout a réussi.
\end{enumerate}

Le skill lit deux workflows de référence (v2) pour reproduire le format XML exact, et génère un fichier \texttt{import\_<primary\_entity>.hwf}.

\subsubsection{Méthode d'amélioration itérative — Perfect Output}

Après la rédaction initiale des skills, une phase d'amélioration itérative a été appliquée pour augmenter leur fiabilité. La méthode consiste à~:

\begin{enumerate}
    \item \textbf{Créer un dataset de test}~: un jeu de données CSV couvrant les cas de figure à vérifier.
    \item \textbf{Créer manuellement les pipelines et le manifeste de référence}~: ce sont les résultats attendus pour ce dataset.
    \item \textbf{Exécuter le skill}~: lancer l'agent sur le dataset et comparer sa sortie avec la référence.
    \item \textbf{Identifier les écarts}~: chaque différence entre la sortie de l'agent et la référence constitue un défaut du skill.
    \item \textbf{Corriger le skill}~: ajouter les règles, conventions ou exemples manquants dans le prompt ou les fichiers de référence.
    \item \textbf{Itérer}~: relancer jusqu'à convergence (sortie = référence).
\end{enumerate}

Cette boucle a permis de découvrir et documenter de nombreuses conventions non évidentes du XML Apache Hop. À chaque itération, les corrections apportées au skill sont vérifiées manuellement pour s'assurer qu'elles restent généralisables et ne sur-optimisent pas pour un cas particulier (overfitting).

\subsubsection{Le mécanisme PTY}

Le choix d'un pseudo-terminal (PTY) plutôt qu'un appel HTTP classique est une décision architecturale centrale. L'outil \texttt{AskUserQuestion} de Claude nécessite un mode interactif : il affiche des choix cliquables dans le terminal et attend une réponse de l'utilisateur. Ce comportement est incompatible avec le mode sans interaction (\texttt{-p}), qui ne permet pas d'entrée utilisateur après le prompt initial.

La solution retenue repose sur \texttt{pty.openpty()}, qui crée une paire maître/esclave : Claude est lancé sur le côté esclave, tandis qu'une boucle de relais lit en continu depuis le maître et transfère les données vers le WebSocket (et inversement). Le frontend reçoit ainsi un flux terminal authentique, rendu par xterm.js dans le navigateur.

\subsubsection{Le registre des entités}

Le fichier \texttt{registry.py} maintient une liste curatée manuellement des clés étrangères obligatoires pour chaque entité. Cette liste ne se limite pas aux champs marqués \texttt{required="true"} dans les schémas XML d'Axelor : certaines FK sont techniquement nullables dans la base mais indispensables en pratique (par exemple, la devise pour toute entité liée à la comptabilité). La curation manuelle garantit que la liste reflète les contraintes métier réelles plutôt que les seules contraintes techniques.
